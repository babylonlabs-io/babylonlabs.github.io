# Vault Creation

Creating a vault is the process of locking Bitcoin on the Bitcoin network and
making it available as collateral on Ethereum. This is called a **peg-in**.

Before any Bitcoin moves, the protocol coordinates multiple participants to
construct a set of pre-signed Bitcoin transactions that encode every possible
outcome -- repayment, liquidation, and fraud challenge. This ensures your BTC
is protected from the start.

You can perform vault creation through the **web portal** or the **CLI**.

## The User Experience

At a high level, creating a vault involves three phases:

### Phase 1: Initiate

You start by submitting a vault creation request on Ethereum and broadcasting
a preliminary Bitcoin transaction. Specifically:

- You declare your intent to create a vault by submitting a request to the
  **BTCVaultRegistry** contract on Ethereum. This includes your Bitcoin public
  key, the amount you want to deposit, and which vault provider you want to
  work with.
- A small **peg-in fee** (in ETH) is required to prevent spam.
- Your deposit amount must meet the protocol's minimum threshold.
- The vault provider must be registered and active.

Once the request is submitted, the protocol emits an event that signals
off-chain participants to begin setup.

### Phase 2: Setup (Automatic)

This phase happens automatically in the background -- you do not need to take
any action beyond signing when prompted. Here is what happens behind the scenes:

1. **Transaction graph construction.** The vault provider and vault keepers
   create a complete set of pre-signed Bitcoin transactions covering every
   possible vault outcome. These participants coordinate via secure channels and
   cross-sign each other's transaction sets.

2. **WOTS keypair setup.** Participants generate Winternitz One-Time Signature
   (WOTS) keypairs for the BitVM3 verification system. This involves setting
   up 6 garbled circuit (GC) instances per counterparty relationship, which
   enable trustless proof verification on Bitcoin later.

3. **Payout signing.** You sign the payout transactions, authorizing the
   conditions under which your BTC can be released. The web portal or CLI
   handles this signing step for you.

4. **Acknowledgment collection.** Each participant submits an on-chain
   acknowledgment (ACK) confirming they hold a complete set of pre-signed
   transactions. Once all ACKs are collected, the vault status moves to
   **Verified**.

This phase typically takes 10-15 minutes.

### Phase 3: Activate

With the vault verified and all transaction graphs in place, you can safely
lock your Bitcoin:

1. **Broadcast your Bitcoin transaction.** You sign and broadcast the peg-in
   transaction, sending your BTC to the vault's Taproot script address on the
   Bitcoin network.

2. **Wait for confirmations.** The protocol waits for your Bitcoin transaction
   to be mined and reach sufficient confirmation depth for reorg protection.
   This is the longest step, typically taking 30-60 minutes.

3. **Activation.** The vault provider submits a Merkle inclusion proof to
   Ethereum, confirming your Bitcoin transaction was included in a Bitcoin
   block. The contract verifies this proof against the on-chain Bitcoin light
   client and confirms that the vault's Taproot output correctly commits to all
   expected participants. The vault status transitions to **Active**.

## After Activation

Once active, your vault can be:

- **Used as collateral** on Aave v4 to borrow stablecoins and other assets.
- **Combined with other vaults** in a single lending position.
- **Left idle** until you are ready to use it -- an active vault does not need
  to be used immediately.

Your Bitcoin remains locked on the Bitcoin network throughout. The accounting
token (vaultBTC) is minted on Ethereum only when you actively deposit the vault
as collateral.

## Timeouts and Refunds

If the peg-in process is not completed within the activation timeout (a protocol
parameter measured in Ethereum blocks), the vault expires. This prevents
indefinite resource locking and protects all participants from being committed
to stale transaction data.

If the vault expires before your Bitcoin transaction is broadcast, no BTC is at
risk since it was never sent. If the vault expires after you broadcast but before
activation, the pre-signed transaction graph includes a refund path that returns
your BTC.

## Timeline Summary

| Phase | What Happens | Approximate Duration |
|-------|-------------|----------|
| Phase 1: Initiate | Submit request on Ethereum | ~1 Ethereum block |
| Phase 2: Setup | Transaction graphs, WOTS setup, signing, ACKs | ~10-15 minutes |
| Phase 3: Activate | Bitcoin broadcast, confirmations, inclusion proof | ~30-60 minutes |
| **Total** | **End-to-end** | **~1 hour** |
