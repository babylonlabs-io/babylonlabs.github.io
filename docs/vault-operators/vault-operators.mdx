# Vault Operators

Run infrastructure for the Trustless Bitcoin Vault protocol.

This section is for teams running vault provider (VP), vault keeper (VK), and
universal challenger (UC) infrastructure. The architectural overview below
reflects the current daemon layout, with command and port references verified
against the public `babylonlabs-io/btc-vault` repository.

## Daemon Architecture

A vault operator runs two cooperating daemons that coordinate to participate in
the protocol:

### `vaultd` — Protocol Coordination

The main daemon that handles protocol-level work:

- Monitors Ethereum events for vault lifecycle changes
- Coordinates BaBe sessions and transaction-graph presigning with peer daemons
- Generates and verifies WOTS commitments and challenge artifacts
- Drives peg-in and peg-out claims, payouts, and challenge responses
- Connects to a separate Bitcoin signing daemon for any operation that needs
  Bitcoin keys
- Stores its operational state in **PostgreSQL**; schema migrations run via
  `vaultd db migrate`

### `btcwd` — Remote BTC Signer

The Bitcoin wallet daemon (`btcwd`) is a separate process that holds the
operator's Bitcoin signing keys and exposes a protocol-agnostic gRPC API to
`vaultd`. It is responsible for:

- Funding and signing claim, assert, payout, and challenge transactions
- Managing UTXOs, fee estimation, and CPFP for fee bumping
- Returning signed PSBTs back to `vaultd`
- Storing wallet state (descriptors, UTXOs, reservations) in a local **SQLite**
  BDK database

Splitting the wallet from `vaultd` provides several operational benefits:

- **Smaller attack surface** — only the wallet daemon needs access to private
  keys; `vaultd` only sees signed transactions
- **HSM / KMS integration** — the wallet can be backed by external signers
  through the same gRPC interface
- **Independent restarts** — wallet outages don't lose protocol session state
- **HMAC-authenticated link** — `vaultd` and `btcwd` authenticate to each
  other using a shared HMAC secret. A single 32-byte secret is loaded by both
  daemons (`btcwd` from `[grpc].hmac_secret_file` or `hmac_secret_env`,
  `vaultd` from its `[btc_wallet]` config block). A wrong or missing secret
  produces immediate `Unauthenticated` errors at connection time.

`btcwd` refuses to bind a non-loopback `listen_addr` (default `127.0.0.1:9091`)
unless `btcwd start --allow-insecure-bind` is set. Keep `btcwd` on loopback
unless you have a deliberate operational reason and out-of-band network
isolation.

## Daemon-to-Daemon Communication

All inter-daemon traffic between vault operators uses gRPC:

- **BaBe setup** between claimers and challengers runs over gRPC streams with
  CWT bearer tokens (CBOR Web Tokens) that automatically refresh during
  long-running sessions. Each `initiate_babe_setup` caller's identity is bound
  to the request parameters so a stolen token cannot be replayed for a
  different session.
- **Transaction-graph presigning** between vault providers and vault keepers
  uses authenticated gRPC RPCs. Universal Challenger PegIn signatures are
  bound to the Pre-PegIn HTLC output to prevent cross-vault replay.
- **Depositor-to-VP requests** (peg-in setup, signature submission) use gRPC
  authenticated with an auth anchor (signed challenge proving control of the
  BTC key) or BIP-322 signature as fallback. The auth anchor produces a CWT
  bearer token used for subsequent requests.

## RPC Server Layout

`vaultd` exposes **two HTTP JSON-RPC listeners** per daemon:

| Listener   | Bind                                              | Methods                                                                | Auth                                                |
|------------|---------------------------------------------------|------------------------------------------------------------------------|-----------------------------------------------------|
| Public RPC | configurable (default `127.0.0.1`, port `8080`)   | depositor + read-only daemon-to-daemon                                 | CWT bearer (BIP-322 anchor for depositors)          |
| Admin RPC  | always `127.0.0.1`                                | full (public + register, resume, unsafe, debug, dev, security-council) | none — loopback-only with DNS-rebinding host filter |

The admin port defaults to **public port + 1000** (`AdminRpcConfig::resolve_port`).
With the typical operator convention of public ports 8080 / 8081 / 8082 for
VP / VK / UC, the admin ports are **9080 / 9081 / 9082**. Set
`[rpc.admin].port` explicitly when running multiple daemons on the same host.

The CLI's operator-facing subcommands (`vp register`, `*p publish-metadata
--daemon-url …`, `vp resume`, `vk resume`, `dev set-prover-mode`, `unsafe-*`)
all target the admin RPC. The `--disable-auth` global flag affects only the
public RPC listener; the admin RPC has no auth and relies entirely on its
loopback bind plus the host filter.

## Vault Provider Operations

Vault providers act as claimers in their own right and as the coordinator that
drives a vault's setup. Detailed setup runbooks (hosted internally) cover:

- **Setup guide** — software installation, configuration, and bootstrapping
  the on-chain registration and metadata
- **`btcwd` configuration** — wallet provisioning (`btcwd init`), HMAC pinning,
  and optional HSM / KMS integration
- **Network and TLS** — gRPC endpoint exposure with optional TLS termination
  through reverse proxies (nginx, etc.)
- **Proof generation** — operating the SP1 prover, GPU capacity planning, and
  prover-side recovery from interrupted proofs
- **Monitoring** — Prometheus metrics, structured tracing, and recommended
  alert thresholds for ETH wallet balance, BaBe session counts, and stuck
  peg-ins
- **Watchtower delegated claims** — `vaultd vp watchtower {start-claim,
  finalize-payout, broadcast-wrongly-challenged}` for one-shot delegated
  claim/payout/dispute operations that read inputs from files and run without
  a live VP daemon

## Vault Keeper Operations

Vault keepers are application-level participants who co-sign the transaction
graph during peg-in and act as challengers and arbitrageurs. Topics:

- **Registration** — application-scoped on-chain registration through
  `BTCVaultRegistry` via `vaultd vk publish-metadata` (online or offline mode),
  BTC key management, and metadata publication
- **Transaction graph storage** — persisting and backing up presigned graphs
  and BaBe artifacts in the PostgreSQL state DB
- **Liquidation settlement** — operating as an arbitrageur for `BTCVaultSwap`
  and other liquidation liquidity providers (LLPs)

## Universal Challenger Operations

Universal challengers are protocol-level monitors that watch for fraudulent
claims and execute challenges. Topics:

- **Deployment** — running the daemon, connecting to Ethereum and Bitcoin
  data sources, and managing challenge bonds
- **Challenge execution** — detecting invalid Assert transactions and
  broadcasting `ChallengeAssert` transactions, with retry on transient
  broadcast errors and `NoPayout` re-verification. UCs do not have a
  `register` or `resume` CLI subcommand; on-chain UC-set membership is
  published via `vaultd uc publish-metadata` (which has `--rpc-url` and
  `--daemon-grpc-url` but no `--grpc-url` — UCs do not run a BaBe server).

## Security Council

Emergency `CouncilNoPayout` transactions are signed offline by a quorum of
council members and combined by a coordinator. The CLI supports the workflow
directly:

- `vaultd security-council sign-nopayout` — each council member produces a
  signed PSBT contribution
- `vaultd security-council aggregate` — the coordinator combines the
  contributions and (optionally) broadcasts the resulting transaction

## Status Monitoring

Operators monitor running peg-ins, peg-outs, and challenges through:

- The peg-in invariants documented in the protocol specification (status
  transitions, retry behavior, expiry handling)
- The vault debugger UI (operator URL provided by your devops contact)
- JSON-RPC `getPeginStatus` / `getPegoutStatus` calls for individual flows
- `btcwd status` (JSON output: balance, available UTXOs, reservations)
- Standard Prometheus metrics emitted by `vaultd` and `btcwd`
