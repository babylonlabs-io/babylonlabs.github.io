# Withdraw & redeem

Redemption (peg-out) is the reverse of vault creation. It unlocks the bitcoin sitting in the vault's Taproot output and sends it back to the depositor's Bitcoin address. This page covers the two paths to redeem, what each step changes on Ethereum and on Bitcoin, and how long the round trip takes.

## Two ways to exit

Two routes to recover BTC:

* **Normal exit.** Repay any outstanding debt, withdraw the vault from the lending position, and let the Vault Provider drive the Bitcoin claim. This is the common path.
* **Depositor self-claim.** Broadcast the Bitcoin claim using the Winternitz One-Time Signature (WOTS) key and claimer artifacts committed at vault creation, via the protocol's `watchtower` CLI tool. Use this if the Vault Provider is offline, slow, or refuses to act.

## Before you start

Check the following on the position:

* **Debt on every borrowed reserve.** A full withdrawal requires zero debt on every reserve borrowed from (USDC, USDT, WBTC).
* **Health factor.** A partial withdrawal is allowed only while the health factor stays above 1.0 after the withdrawal.
* **BTC Vault status.** Each vault must be either `InUse` (still in the position) or `Available` (already withdrawn from the position).

The application surfaces all three of these on the Loans card.

## Step 1: Repay debt (Aave v4 application)

If there is any outstanding debt, repay it before attempting a full withdrawal. For each borrowed reserve:

1. Approve the AaveAdapter to spend the repayment token (standard ERC-20 approval).
2. Repay the debt with required token.
3. Clear debt on every reserve ever borrowed from.

## Step 2: Withdraw collateral and start redemption

Open the Collateral card and click **Withdraw**.

* Select the vault to withdraw.
* Review the withdrawal amount, network fee rate, VP commission, and the nominated Bitcoin address.
* Confirm the Ethereum transaction.

**Partial withdrawal.** Pass a subset of vault IDs. The contract verifies the health factor stays above 1.0 after the listed vaults are removed. If removing them would push the position into liquidatable territory, the transaction reverts.

**Full withdrawal.** Pass every vault in the position. The contract verifies that all debt (principal and interest) is fully repaid on every borrowed reserve.

## Step 3: Wait through the challenge period (TBV protocol)

After redemption starts on Ethereum, the Vault Provider drives the Bitcoin side. Bitcoin cannot natively verify Ethereum state, so the protocol uses a ZK proof of the burn event, verified on Bitcoin via Babylon's BABE construction. This step is protocol-level; the same flow applies regardless of which application released the vault. The flow on Bitcoin runs in four steps:

1. **Claim.** The Vault Provider broadcasts a claim transaction that spends from the vault's Taproot output. (Automatic; no depositor action needed.)
2. **Assert.** The Vault Provider broadcasts an assert transaction containing the ZK proof. (Automatic; proof generation runs on the prover service.)
3. **Challenge window.** An Application Vault Keeper or Universal Challenger may dispute the claim during this window, which runs approximately **3 days**. In the happy path no challenge is raised. This is a passive wait.
4. **Payout.** Once the window closes, the Vault Provider broadcasts the payout, releasing BTC to the depositor's Bitcoin address. The Vault Provider's commission (set per-VP, embedded in the pre-signed payout transactions) is deducted at this step.

## The self-claim path (TBV protocol)

The Vault Provider drives most redemptions. The depositor does not depend on the Vault Provider to recover BTC; the protocol exposes a self-claim path that is always invocable. Self-claim is a protocol-level feature, available to depositors of any application built on TBV.

During vault creation, the depositor committed a WOTS public key hash on Ethereum, shared the full WOTS public key with the Vault Provider, and downloaded a set of claimer artifacts (transaction graph, BABE session data per challenger, verifying key). The Vault Provider generated a parallel set of pre-signed Bitcoin transactions naming the depositor as authorized claimer. The PegIn transaction also produced a small "depositor claim output" controlled solely by the depositor's Bitcoin key. All of these sit dormant unless they are needed.

### When to self-claim

* The Vault Provider is offline, slow, or unresponsive.
* The Vault Provider refuses to act on the redemption.
* The depositor wants to drive the Bitcoin claim themselves rather than wait.

### What is needed

* **WOTS keypair file** for the specific vault being redeemed. Each vault has its own WOTS key; that key can be used exactly once. The application stores the file alongside the wallet; export it from the application's keys section for offline backup.
* **Claimer artifacts** downloaded at peg-in: transaction graph JSON, verifying key, and per-challenger BABE session artifacts.
* **The `watchtower` CLI tool:** a command-line program the protocol ships for the self-claim path. It is idempotent (commands can be retried safely) and no long-running daemon is required.

### Backups and recovery

The self-claim path is the protocol's bottom-line safety net. Back up both inputs at peg-in:

* Export the WOTS keypair file from the application and store it alongside the seed phrase. Each vault needs its own WOTS file.
* Download the claimer artifacts and store them somewhere that will still be accessible in 6 months. Cold storage is acceptable.

If only one of the WOTS file or the artifacts is lost (not both), self-claim is no longer possible for that vault, but the standard path (Vault Provider drives the claim) still works; the self-claim fallback is only needed if the VP is also unresponsive.

If both the WOTS file and the artifacts are lost, and the Vault Provider is also unresponsive, recovery requires Security Council intervention.

## What happens to the internal accounting

The internal accounting token vaultBTC exists only while the BTC is supplying a lending position.

* When a vault is withdrawn, the corresponding vaultBTC is burned in the same transaction.
* Total vaultBTC supply always equals BTC currently held as active collateral across all positions in the Aave Adapter.
* vaultBTC cannot be transferred to an arbitrary address. It moves only between authorized protocol contracts and does not exist on any secondary market.

## Timeline summary

| Stage | Layer | Where | Duration |
| --- | --- | --- | --- |
| Repay debt on each borrowed reserve | Aave v4 application | Ethereum | Minutes |
| Withdraw collateral; start redemption | App to protocol handoff | Ethereum | One transaction |
| Bitcoin claim and assert broadcasts | TBV protocol | Bitcoin | Minutes to hours |
| Challenge window | TBV protocol | Bitcoin | ~3 days |
| Payout to the depositor's BTC address | TBV protocol | Bitcoin | Confirms once mined |

The Ethereum-side actions complete in minutes. The Bitcoin-side challenge window dominates wall-clock time.
