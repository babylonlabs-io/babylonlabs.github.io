# TBV Glossary

Terms used throughout the documentation. Each entry links to the page where the concept is expanded.

**AaveAdapter.** An Ethereum smart contract for TBV-on-Aave operations. Coordinates adding vaults as collateral, borrowing, repaying, withdrawing, and triggering redemption. See [Aave v4 integration](../technical-details/aave-v4-integration.mdx).

**Aave v4 Hub.** Central liquidity contract in Aave v4 from which spokes draw borrowable assets. Each asset is listed on the Hub with its own interest-rate strategy and other risk parameters, and lenders supply listed assets to the Hub to earn interest. Spoke contracts receive per-asset draw and add caps for every asset they support. The Babylon Core Spoke draws from the Hub when a depositor borrows; repays and liquidations restore liquidity. See [Aave v4 integration](../technical-details/aave-v4-integration.mdx).

**Application Vault Keeper (AVK).** An application-scoped Vault Keeper that holds redemption rights over a vault. Under defined conditions it can claim the vault, and it can challenge any invalid claim made by other claimers, so it is both a claimer and a challenger. Each application names this role differently; in the Aave v4 integration the Application Vault Keeper is the arbitrageur. See [TBV Protocol actors](../technical-details/protocol-actors.mdx).

**ApplicationRegistry.** Ethereum contract listing the DeFi applications approved to receive vaults as collateral. Currently the only registered application is the Aave v4 integration. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**BABE.** Proof-verification construction used by the protocol. A cut-and-choose protocol establishes the challenge framework between each claimer-challenger pair at vault creation, generating many candidate garbled-circuit instances for dispute resolution. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**Babylon Core Spoke.** A dedicated, isolated Aave v4 market that accepts vaultBTC as collateral. Has its own risk parameters (collateral factor, liquidation bonus, target health factor) and draws borrowable assets from the Aave v4 Hub. See [Aave v4 integration](../technical-details/aave-v4-integration.mdx).

**BitVM3.** Bitcoin-side construction family that lets a zero-knowledge proof generated off-chain be verified by Bitcoin's existing script capabilities, without requiring a Bitcoin soft fork. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**BTCVaultSwap.** The first Liquidation Liquidity Provider registered for the Aave v4 integration; a smart contract. It lets a seized vault be swapped for a fungible asset so that liquidators on the permissionless path can be paid instantly. The fungible payout asset can be any fungible wrapped-BTC token. A registered Application Vault Keeper (the arbitrageur for the Aave v4 integration) later acquires the escrowed vault, repays the Hub debt, and finalizes the BTC redemption on the Bitcoin Network. See [Aave v4 integration](../technical-details/aave-v4-integration.mdx).

**CapPolicy.** Ethereum contract enforcing per-application and per-address BTC exposure caps. At vault activation, the contract checks that the activation would not exceed either cap. See [Protocol parameters](../testnet-info/protocol-parameters.mdx).

**Challenge period.** A window during which any Vault Keepers (Vault Providers or Application Vault Keepers) or Universal Challengers can dispute a claim on the Bitcoin Network. The depositor can also dispute through their Vault Provider. If no challenge arrives before the window closes, the claim finalizes and the BTC is released to the claimer. See [How it works](../start-here/how-it-works.mdx).

**Cliff effect.** The risk that a single-vault position is fully seized at liquidation because each vault is indivisible. Splitting a depositor's BTC into a sacrificial vault and a protected vault at peg-in mitigates this. See [Liquidation risk](../use-for-lending/liquidation-risk.mdx).

**Collateral factor.** The fraction of a collateral asset's value counted toward borrowing capacity. With a collateral factor of 78% (the public-testnet value for vaultBTC), $60,000 of BTC supports up to $46,800 of borrows. Set per asset on the Babylon Core Spoke. See [Borrow & repay](../use-for-lending/borrow-and-repay.mdx).

**Depositor.** The Bitcoin holder who creates a vault to use BTC as collateral. The depositor's keys are required at vault creation and at every release path. Also called the borrower in lending contexts. See [TBV Protocol actors](../technical-details/protocol-actors.mdx).

**Depositor claim output.** A small Bitcoin output produced by the PegIn transaction (Output 1) controlled solely by the depositor's key. Serves as the structural anchor for the depositor's self-claim path during peg-out: if the Vault Provider goes offline, the depositor can spend this output as part of the independent claim. See [Withdraw & redeem](../use-for-lending/withdraw-and-redeem.mdx).

**Fairness debt repay.** Compensation mechanism during partial-position liquidation when the over-seizure surplus is less than the depositor's remaining debt: the surplus reduces the depositor's debt rather than being paid out in WBTC. See [Liquidation risk](../use-for-lending/liquidation-risk.mdx).

**Fairness payment.** Compensation paid to the depositor in WBTC on full-position liquidation when all debt is covered by the seizure and a surplus remains. The over-seizure value is discounted using the ratio of debt liquidated to collateral liquidated, not valued at raw BTC price. WBTC is the denomination because the seized vault's BTC has already moved to the liquidator on Bitcoin. The complementary mechanism for partial-position liquidation is fairness debt repay. See [Liquidation risk](../use-for-lending/liquidation-risk.mdx).

**Garbled circuit (GC).** A cryptographic construction used in the BABE construction to evaluate complex predicates on the Bitcoin network. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**Hashlock.** The SHA-256 hash of a 32-byte secret generated by the depositor at peg-in. The hash is committed on Ethereum; the secret stays with the depositor until activation. Revealing the secret on Ethereum unlocks the matching Bitcoin transaction. See [How it works](../start-here/how-it-works.mdx).

**Health factor.** Ratio of risk-adjusted collateral value to total debt value. A position is liquidatable when the health factor drops below 1.0. See [Liquidation risk](../use-for-lending/liquidation-risk.mdx).

**Liquidation bonus.** A flat 10% discount on seized collateral granted to the liquidator as an incentive to keep the protocol solvent. See [Aave v4 integration](../technical-details/aave-v4-integration.mdx).

**LLP (Liquidation Liquidity Provider).** A contract that provides instant fungible-asset liquidity (for example a wrapped-BTC token) to liquidators while the underlying BTC redemption finalizes asynchronously. See [Aave v4 integration](../technical-details/aave-v4-integration.mdx).

**Partial-position liquidation.** When a position holds multiple vaults and only the minimum subset of vaults needed to restore the health factor gets seized; the rest remain in the position. The protocol walks vaults in their stored order during liquidation. See [Aave v4 integration](../technical-details/aave-v4-integration.mdx).

**Peg-in / peg-out.** Peg-in is the process of locking BTC into a vault on Bitcoin and registering the vault on Ethereum. Peg-out is the reverse: a vault is redeemed and the BTC is released back to a Bitcoin address. See [How it works](../start-here/how-it-works.mdx).

**Protected vault.** In a two-vault position, the vault placed second in the liquidation order. Designed to remain untouched during partial-position liquidation while the sacrificial vault is seized. See [Liquidation risk](../use-for-lending/liquidation-risk.mdx).

**Sacrificial vault.** In a two-vault position, the vault placed first in the liquidation order. Sized to cover the protocol's expected seizure amount so that only this vault is seized if the position becomes liquidatable. See [Liquidation risk](../use-for-lending/liquidation-risk.mdx).

**SP1 proof.** A zero-knowledge proof system used by the Claimer to generate the proof that an Ethereum burn event occurred. Run by an SP1 prover (locally or via an external prover service), typically on GPU hardware. The proof is then aggregated to Groth16 form and verified on Bitcoin. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**Target health factor (THF).** The health factor the protocol aims to restore after partial-position liquidation. Set per-spoke by governance. Used to compute how many vaults to seize. See [Aave v4 integration](../technical-details/aave-v4-integration.mdx).

**Taproot script.** A Bitcoin output type with multiple spending paths encoded as separate leaves in a Merkle tree. Each vault's BTC sits in a Taproot address whose leaves include withdrawal (post-repayment), liquidation, fraud-challenge response, and Pre-PegIn refund paths. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**Transaction graph.** The set of pre-signed Bitcoin transactions covering every legitimate spending path for a vault. Constructed and signed by all required participants at vault creation. After creation, no participant can fabricate a new spend. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**UTXO.** Unspent Transaction Output. Bitcoin's accounting model: each piece of bitcoin in circulation is a UTXO, indivisible, that gets fully consumed when spent. Each vault corresponds to exactly one UTXO. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**Vault Provider commission.** The percentage of BTC a Vault Provider takes from a vault's redemption payout. Set per-VP and embedded in the pre-signed Payout transactions at vault creation; depositors accept the VP's commission rate when they choose the VP. See [TBV Protocol actors](../technical-details/protocol-actors.mdx).

**WOTS (Winternitz One-Time Signature).** Each vault commits a WOTS public key hash at peg-in; the corresponding private key can only be used once and is the depositor's authorization for the self-claim path. See [Protocol architecture](../technical-details/protocol-architecture.mdx).

**ZK proof.** A cryptographic proof that a statement is true without revealing why. The protocol uses ZK proofs to attest, on Bitcoin, that specific Ethereum events (such as a vault redemption) occurred on the finalized Ethereum chain. Verified on Bitcoin via the BABE construction. See [How it works](../start-here/how-it-works.mdx).
