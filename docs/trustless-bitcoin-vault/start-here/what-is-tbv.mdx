# What is TBV?

TBV is currently running on Bitcoin signet and Ethereum testnet. Funds involved are test-only and have no monetary value. For mainnet plans, see [Audits & security](../reference/audits-and-security) and [Community & support](../reference/community-and-support).

## The problem: only ~1% of Bitcoin is used in DeFi

The largest asset in crypto sits mostly idle. The reason is structural: the Bitcoin Network has no general smart-contract execution, and DeFi lives on chains that do. To bring BTC into DeFi today, the holder has to give the BTC to a custodian (who issues a BTC token on another chain), trust a cross-chain bridge, or accept a federation of signers as a substitute for self-custody. Every existing path replaces Bitcoin's trustless settlement with a new trust assumption.

**Trustless Bitcoin Vault (TBV)** is Babylon's protocol for closing this gap. The depositor's BTC stays on the Bitcoin Network, held in a Taproot script that the depositor co-signs at creation. An Ethereum-side coordination contract tracks each vault and lets a DeFi application use it as collateral. Cross-chain state moves both ways through cryptography, not through an intermediary:

* An on-Ethereum **Bitcoin light client** (implemented as smart contracts) verifies the state of Bitcoin-side vaults so Ethereum applications can act on Bitcoin events.
* A **BABE construction** lets Bitcoin verify zero-knowledge proofs of Ethereum events using only existing Bitcoin script primitives. No Bitcoin fork is required.

The trust assumption is the protocol's cryptography, not an intermediary's solvency.

## What a vault is — and what it is not

In DeFi the word "vault" usually means a capital pool: many users deposit into one shared contract and share returns or risks. A Trustless Bitcoin Vault returns to the original meaning of "vault" — closer to the secure compartment in a bank than to a DeFi capital pool. Each vault is a **segregated**, depositor-owned Bitcoin output. Three properties define it:

* **The BTC is locked in a Bitcoin script that the depositor created.** Every legitimate spending path — repayment, liquidation, fraud challenge, refund — is signed at vault creation by the depositor and the protocol participants. After that, no party can fabricate a new spend.
* **BTC in different vaults is segregated.** A vault is one UTXO; one depositor; one set of pre-signed exit paths. Vaults are not pooled.
* **No one can rehypothecate the BTC.** The protocol contracts cannot move the BTC out of the script, lend it elsewhere, or repurpose it. The collateral does only what the script allows.

## The trust model, before and after

Bringing Bitcoin into DeFi using a custodian or a bridge requires the depositor to trust at least one party beyond the chains themselves: the custodian who holds the BTC, or the operators of the bridge. A custodian compromise or a bridge exploit severs the depositor's claim on the original BTC.

With TBV, the trust set is shorter:

* Trust in the **Bitcoin Network** and in the Bitcoin script the depositor co-signed at vault creation.
* Trust in the **Ethereum network** and in the target DeFi smart contract.

Beyond the chains themselves, residual trust sits with the protocol's governance and emergency-response multi-sigs — transitional safety nets that the protocol can retire over time. There is no custodian, no bridge contract holding the BTC, and no federation of signers required to release funds. The depositor's own keys participate at every depositor-controlled release path, and a self-claim path is committed at vault creation so the depositor can exit independently if the Vault Provider becomes unresponsive. See [Safety & trust assumptions](./safety-and-trust-assumptions) for the full picture.

## Two layers: the TBV protocol and the DeFi applications on top

* **The TBV protocol.** The core, application-agnostic protocol. It owns vault creation (peg-in), vault redemption (peg-out), the BABE fraud-detection mechanism, the Bitcoin light client on Ethereum, and the vaultBTC accounting token. Vaults exist independently of any particular DeFi application — a vault can be supplied as collateral to any registered application over its lifetime.
* **DeFi applications on top.** Each registered consumer of vaultBTC. An application defines its own product — lending, stablecoins, options, insurance — and is responsible for its own borrowing, liquidation, and reward logic. The first application is **Aave v4 lending**.

## What you can test today

On the open Testnet, a depositor with an active vault can **test** borrowing stablecoins (USDC, USDT) or WBTC against it. The borrowed funds and the underlying BTC are signet and testnet assets with no monetary value — this is an **experiment** in trustless Bitcoin DeFi, end to end. After repayment, the vault is unlocked and the BTC is released back to the depositor's Bitcoin address.

Multiple vaults can combine into a single borrowing position. The app recommends splitting BTC into two vaults so that if the position faces a liquidation event, only the first vault is seized and the rest stays as collateral. The mechanism is explained on [Liquidation risk](../use-for-lending/liquidation-risk).
